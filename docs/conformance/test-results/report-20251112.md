## Technical Implementation Notes

### Architecture Decisions

1. **Token Revocation Strategy:**
   - Use REVOKED_TOKENS KV namespace for blacklist approach
   - Store JTI with same TTL as token expiration
   - Mark authorization codes as used instead of deleting (detect reuse)

2. **Claims Parameter Implementation:**
   - Parse claims JSON in authorization endpoint
   - Store in authorization code data
   - Pass through to access token
   - UserInfo endpoint checks both scope and claims parameter

3. **Scope-Based vs Claims-Based Access:**
   - Scope grants ALL related claims (standard OIDC behavior)
   - Claims parameter allows individual claim requests
   - When both present, scope takes precedence (returns all claims)

4. **Address Scope Format:**
   - Returns address as object (OIDC Core 5.1.2)
   - Includes: formatted, street_address, locality, region, postal_code, country

## Known Issues & Limitations

### Current Limitations (By Design for Phase 3)

1. **Static User Data:**
   - UserInfo endpoint returns hardcoded test data
   - Production implementation will require database integration (Phase 6)

2. **No User Authentication:**
   - Authorization endpoint automatically approves all requests
   - No login UI or consent screen (Phase 6)

3. **No Session Management:**
   - Cannot track user authentication state
   - prompt, max_age, id_token_hint parameters not supported (Phase 6)

4. **No Refresh Token:**
   - Only authorization code flow supported
   - Refresh token flow planned for Phase 4

5. **Single Key:**
   - No key rotation mechanism yet
   - Key rotation planned for Phase 4 (Durable Objects)

### Environment-Specific Considerations

1. **KV Eventual Consistency:**
   - Cloudflare KV is eventually consistent
   - Code reuse detection may have small race condition window
   - Acceptable for MVP, will monitor in production

2. **Edge Deployment:**
   - Global distribution via Cloudflare Workers
   - Low latency for authorization requests
   - KV access adds ~5-50ms depending on region
